AutoKeep (跨平台架构版)
1. 跨平台技术架构 (Future-Proof Design)
核心框架： React + Vite (Web 端的基石)。
跨平台容器： Capacitor (这是关键！它允许你的 Web 代码直接调用手机原生功能，如通知、摄像头、文件系统)。
UI 框架： Tailwind CSS + Headless UI / Shadcn UI (必须保证响应式，一套 UI 同时适配网页和手机屏幕)。
后端 API： Supabase SDK (天然支持多端，无论是 Web、Android 还是 iOS，调用接口的代码一模一样)。
2. 数据库与接口预留 (API-First)
由于 Supabase 本身就是通过 RESTful API 通讯的，所以你的接口已经是“预留”好的。

Web 端： 通过 supabase-js 调用。
未来 Android 端： 可以继续用 supabase-js（如果是 Capacitor 方案），或者用 supabase-kt（如果是纯原生 Kotlin 开发）。数据完全同步。
3. 详细开发路线图 (针对多端优化)
第一阶段：响应式 Web 基石 (当前重点)
UI 设计策略： 采用 "Mobile First"。先在浏览器里把窗口缩到手机大小进行开发。
适配 GitHub Pages：
使用 HashRouter 而不是 BrowserRouter（防止 GitHub Pages 刷新后 404 的经典问题）。
配置 Vite 的 base 路径，确保资源加载正确。
第二阶段：Capacitor 集成 (为 Android 铺路)
在项目中初始化 Capacitor：npx cap init。
这会生成一个 android 目录，你的 Web 代码编译后会被“填入”这个目录。
预留功能： 在代码中封装一个 NotificationService。
Web 版： 调用浏览器的 Notification API。
Android 版： 自动切换为 Capacitor 的 Local Notifications 原生插件。
第三阶段：车型库与数据结构
静态数据分离： 将“宝马油底壳”等常见故障预设放在独立的 JSON 文件或 Supabase 的公共表中。
多设备同步逻辑： 确保当你在网页上更新了里程，Android App 打开时能通过 Supabase 的 Realtime 功能实现实时刷新。
4. 给 Gemini 3.0 的开发指令 (可以直接喂给它)
"你好 Gemini，我们要开发 AutoKeep 的 Web 版，并要求架构具备良好的 Android 扩展性。请执行以下操作：

使用 Vite + React 初始化项目，并引入 Tailwind CSS。
集成 Supabase SDK 作为唯一数据源。
UI 要求： 严格遵循 Mobile-First 响应式设计，确保在手机浏览器上像原生 App 一样流畅。
路由要求： 使用 HashRouter 以适配 GitHub Pages 部署环境。
代码组织： 将所有的 API 请求逻辑封装在 src/services/ 目录下，与 UI 组件完全解耦，方便未来 Android 端复用。"


初始化前端项目结构（项目的“骨架”）
当你导入 Antigravity 编辑器后，你可以要求 Gemini 3.0 按照以下目录结构初始化项目。这个结构预留了 Android 开发的扩展空间：

<TEXT>
/auto-keep
├── src
│   ├── api              # Supabase 接口封装层 (预留 Android 共享逻辑)
│   ├── components       # 通用 UI 组件 (Shadcn UI)
│   ├── hooks            # 自定义 Hook (如 useVehicle, useMaintenance)
│   ├── pages            # 页面 (Dashboard, VehicleDetail, AddLog)
│   ├── store            # 状态管理 (Zustand 或 Context)
│   ├── utils            # 计算逻辑 (核心：保养过期算法)
│   └── App.jsx          # 路由配置 (使用 HashRouter 适配 GitHub Page)
├── supabase             # 存放 SQL 备份和 Edge Functions
├── capacitor.config.ts  # Capacitor 配置 (为 Android 预留)
└── tailwind.config.js   # 样式配置 (Mobile-First)
